# PRD: 수집/전처리 (D-01 ~ D-03)

## 1. 목적

산업단지 내 제조 기업 사원들이 자사 문서(업무 매뉴얼, 품질 기준서, 작업 지시서 등)를 AX 에이전트 시스템에 업로드하면, 다양한 포맷의 문서를 자동으로 파싱·청킹·메타데이터 태깅하여 후속 온톨로지 구축 및 검색 파이프라인에 즉시 활용할 수 있는 상태로 변환하는 것을 목적으로 한다.

### 1-1. 범위

| WBS ID | 기능명 | 요약 |
| --- | --- | --- |
| D-01 | 멀티 포맷 파싱 | PDF, DOCX, HWP, XLSX, CSV의 텍스트·표·이미지 캡션 추출 (OCR 연동) |
| D-02 | 의미론적 청킹 | 문맥(Context) 단위 텍스트 분할로 의미 보존 (Semantic Chunking) |
| D-03 | 메타데이터 추출 | 생성일, 작성자, 문서 분류 등 기본 속성 자동 태깅 |

### 1-2. 용어 정의

| 용어 | 정의 |
| --- | --- |
| 파싱(Parsing) | 비정형 문서 파일에서 구조화된 텍스트·표·이미지 정보를 추출하는 과정 |
| 청킹(Chunking) | 추출된 텍스트를 검색·임베딩에 적합한 단위로 분할하는 과정 |
| 의미론적 청킹(Semantic Chunking) | 고정 길이가 아닌 문맥·주제 경계를 기준으로 분할하는 방식 |
| OCR | 이미지·스캔 문서에서 텍스트를 인식·추출하는 기술 |
| 메타데이터 | 문서 본문 외에 문서를 설명하는 속성 정보(작성자, 생성일, 분류 등) |

## 2. 사용자 및 이해관계자

### 2-1 주요 사용자

| 사용자 | 역할 | 주요 니즈 |
| --- | --- | --- |
| 제조 기업 사원 (일반 사용자) | 문서 업로드 및 질의 | 별도 변환 작업 없이 보유 문서를 그대로 업로드하면 검색 가능해야 함 |
| 도메인 관리자 | 워크스페이스·전처리 설정 관리 | 문서 처리 현황 모니터링, 실패 건 재처리, 전처리 옵션 조정 |
| 시스템 관리자 | 인프라·파이프라인 운영 | 파이프라인 상태 확인, 리소스 모니터링, 장애 대응 |

### 2-2 이해관계자

- 그래프 구축 모듈 (후속 파이프라인): 전처리 출력물을 입력으로 사용
- 검색 모듈: 청킹된 텍스트와 메타데이터를 인덱싱에 활용
- 운영 모듈: 토큰/비용 모니터링, 파이프라인 상태 관리

## 3. 기능 상세 요구사항

### D-01: 멀티 포맷 파싱

#### 1) 개요

제조 현장에서 사용되는 다양한 포맷의 문서를 단일 파이프라인으로 수용하여, 텍스트·표·이미지 캡션을 구조화된 형태로 추출한다.

#### 2) 지원 포맷 및 추출 항목

| 포맷 | 텍스트 | 표(Table) | 이미지 캡션 | 비고 |
| --- | --- | --- | --- | --- |
| PDF | O | O | O | 스캔 PDF는 OCR 연동 필수 |
| DOCX | O | O | O | MS Word 2007+ |
| HWP | O | O | O | 한글 문서 (제조 현장 다수 사용) |
| XLSX | O | O (시트 전체) | - | 셀 병합·수식 결과값 추출 |
| CSV | O | O (전체 데이터) | - | UTF-8 / EUC-KR 인코딩 지원 |

#### 3) 상세 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| D-01-01 | 사용자가 파일을 업로드하면 확장자 및 MIME 타입 기반으로 포맷을 자동 감지한다 | P0 |
| D-01-02 | PDF 내 텍스트 레이어가 없는 경우(스캔 문서) OCR 엔진을 자동 호출하여 텍스트를 추출한다 | P0 |
| D-01-03 | 표(Table)는 행·열 구조를 유지한 Markdown 또는 JSON 형태로 추출한다 | P0 |
| D-01-04 | 이미지에 포함된 캡션 텍스트를 추출하고, 이미지 위치 정보(페이지, 좌표)를 함께 저장한다 | P1 |
| D-01-05 | HWP 파일의 표, 글상자, 각주 등 특수 객체를 정상 파싱한다 | P0 |
| D-01-06 | XLSX의 셀 병합, 수식 결과값, 다중 시트를 각각 분리 추출한다 | P1 |
| D-01-07 | 단일 파일 최대 용량 100MB, 페이지 수 500페이지까지 지원한다 | P0 |
| D-01-08 | 파싱 실패 시 실패 사유(비밀번호 보호, 손상 파일 등)를 명확히 반환한다 | P0 |
| D-01-09 | 배치 업로드(다건 동시 업로드)를 지원하며, 건별 처리 상태를 개별 추적한다 | P1 |

#### 4) OCR 연동 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| D-01-OCR-01 | 한국어·영어 혼합 문서에 대해 95% 이상의 문자 인식률을 보장한다 | P0 |
| D-01-OCR-02 | 표 이미지에 대해 구조 인식(Table Structure Recognition)을 수행하여 행·열을 복원한다 | P1 |
| D-01-OCR-03 | OCR 처리 시 원본 이미지와 인식 결과를 함께 저장하여 사후 검증을 지원한다 | P1 |
| D-01-OCR-04 | 기울어진 스캔, 저해상도 이미지에 대한 전처리(디스큐, 해상도 보정)를 수행한다 | P2 |

### D-02: 의미론적 청킹 (Semantic Chunking)

#### 1) 개요

파싱된 텍스트를 단순 고정 길이가 아닌 문맥(Context) 단위로 분할하여, 임베딩 및 검색 시 의미가 끊기지 않도록 보존한다.

#### 2) 상세 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| D-02-01 | 문장 간 의미 유사도(Embedding Similarity)를 기반으로 주제 전환 지점에서 청크를 분할한다 | P0 |
| D-02-02 | 기본 청크 크기 목표: 300~800 토큰, 최대 1,200 토큰 (설정 가능) | P0 |
| D-02-03 | 인접 청크 간 오버랩(Overlap) 구간을 설정할 수 있다 (기본값: 50~100 토큰) | P0 |
| D-02-04 | 표(Table)는 하나의 독립 청크로 처리하되, 표가 지나치게 클 경우 행 단위로 분할한다 | P1 |
| D-02-05 | 문서 내 계층 구조(장·절·항)를 인식하여 상위 헤딩 정보를 각 청크에 컨텍스트로 부여한다 | P0 |
| D-02-06 | 이미지 캡션은 인접 텍스트와 함께 하나의 청크로 묶는다 | P1 |
| D-02-07 | 청킹 결과에 청크 순서(sequence), 원본 페이지, 원본 문서 ID를 포함한다 | P0 |
| D-02-08 | 도메인별(워크스페이스별) 청킹 파라미터를 독립적으로 설정할 수 있다 | P2 |

#### 3) 청킹 전략

```
[문서 원문]
  │
  ▼
1단계: 구조 기반 1차 분할
  - 장/절/항 헤딩 기준 섹션 분리
  - 표, 이미지 캡션 등 특수 블록 분리
  │
  ▼
2단계: 의미 유사도 기반 2차 분할
  - 섹션 내 문장 간 임베딩 유사도 계산
  - 유사도 급감 지점(Breakpoint)에서 분할
  - 목표 토큰 범위(300~800) 내 조정
  │
  ▼
3단계: 오버랩 적용 및 컨텍스트 부여
  - 인접 청크 간 오버랩 구간 생성
  - 상위 헤딩 정보를 prefix로 부여
  - 청크 메타데이터(순서, 페이지, 문서ID) 태깅
```

### D-03: 메타데이터 추출

#### 1) 개요

문서의 본문 외 속성 정보(작성자, 생성일, 문서 분류 등)를 자동 추출·태깅하여, 필터링 검색 및 문서 관리에 활용한다.

#### 2) 추출 대상 메타데이터

| 분류 | 필드명 | 추출 방법 | 우선순위 |
| --- | --- | --- | --- |
| **기본 속성** | 파일명 | 파일 시스템 | P0 |
|  | 파일 포맷 | 확장자 + MIME | P0 |
|  | 파일 크기 | 파일 시스템 | P0 |
|  | 총 페이지 수 | 파싱 결과 | P0 |
| **문서 속성** | 작성자 | 파일 내장 메타데이터 | P0 |
|  | 생성일 | 파일 내장 메타데이터 | P0 |
|  | 최종 수정일 | 파일 내장 메타데이터 | P0 |
|  | 문서 제목 | 파일 내장 메타데이터 / 본문 첫 헤딩 | P0 |
| **자동 분류** | 문서 유형 | LLM 기반 자동 분류 | P1 |
|  | 키워드 | LLM 기반 핵심 키워드 추출 (5~10개) | P1 |
|  | 요약 | LLM 기반 1~2문장 요약 | P1 |
| **제조 도메인** | 관련 공정 | LLM 기반 / 사전 매핑 | P2 |
|  | 관련 설비 | LLM 기반 / 사전 매핑 | P2 |
|  | 문서 버전 | 파일명 패턴 또는 본문 추출 | P1 |

#### 3) 상세 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| D-03-01 | 파일 내장 메타데이터(작성자, 생성일, 수정일, 제목)를 자동 추출한다 | P0 |
| D-03-02 | 내장 메타데이터가 없는 경우 본문 분석(첫 페이지 헤딩, 날짜 패턴 등)으로 대체 추출한다 | P1 |
| D-03-03 | LLM을 활용하여 문서 유형을 자동 분류한다 (매뉴얼, 기준서, 작업지시서, 보고서, 회의록 등) | P1 |
| D-03-04 | 문서 유형 분류 체계는 도메인(워크스페이스)별로 커스터마이징 가능하다 | P2 |
| D-03-05 | 추출된 메타데이터에 대해 사용자가 수동 보정(편집)할 수 있는 UI를 제공한다 | P1 |
| D-03-06 | 동일 파일명의 문서가 재업로드될 경우 버전 관리(신규 버전 생성 또는 덮어쓰기)를 지원한다 | P1 |
| D-03-07 | 메타데이터 필드를 기반으로 검색 필터링이 가능하도록 인덱싱한다 | P0 |

## 4. 비기능 요구사항

### 4-1. 성능

| 항목 | 기준 |
| --- | --- |
| 단일 문서 파싱 소요 시간 | 50페이지 기준 30초 이내 (OCR 미포함) |
| OCR 포함 파싱 시간 | 50페이지 기준 120초 이내 |

| 배치 업로드 | 1회 최대 50파일 지원 |

### 4-2. 안정성

| 항목 | 기준 |
| --- | --- |
| 파싱 성공률 | 정상 파일 기준 99% 이상 |
| 파이프라인 가용성 | 99.5% 이상 (월간 다운타임 3.6시간 이내) |
| 실패 시 재처리 | 자동 재시도 최대 3회, 이후 수동 재처리 지원 |
| 데이터 유실 방지 | 원본 파일 별도 보관, 처리 중 장애 시 중간 상태 복구 |

| 항목 | 기준 |
| --- | --- |
| 파일 저장 | 업로드 파일 암호화 저장 (AES-256) |
| 접근 제어 | 워크스페이스 단위 문서 격리, 타 워크스페이스 문서 접근 불가 |
| 전송 보안 | 파일 업로드 시 TLS 1.2 이상 적용 |
| 감사 로그 | 문서 업로드·삭제·재처리 이력 기록 |

## 5. 전처리 파이프라인 전체 흐름

```
┌─────────────┐
│  파일 업로드   │  사용자가 UI에서 문서 업로드
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 포맷 감지     │  확장자 + MIME 타입 기반 자동 판별
└──────┬──────┘
       │
       ▼
┌─────────────┐    ┌──────────┐
│ 멀티 포맷     │───▶│   OCR    │  스캔 문서 감지 시 자동 호출
│ 파싱 (D-01)  │◀───│  엔진     │
└──────┬──────┘    └──────────┘
       │
       │  파싱 결과 (텍스트 + 표 + 이미지 캡션)
       ▼
┌─────────────┐
│ 메타데이터     │  파일 내장 속성 + LLM 기반 자동 분류
│ 추출 (D-03)  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 의미론적      │  구조 기반 1차 분할 → 유사도 기반 2차 분할
│ 청킹 (D-02)  │  → 오버랩 적용 → 컨텍스트 부여
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 저장 및 인덱싱 │  청크 + 메타데이터 저장 → 후속 모듈 전달
└──────┬──────┘
       │
       ├──▶ 그래프 구축 모듈 (C-01 ~ C-05)
       ├──▶ 임베딩 / 벡터 인덱스
       └──▶ 검색 모듈 (R-01 ~ R-04)
```

## 7. 연관 모듈 인터페이스

### 7-1 후속 모듈 전달 데이터

| 수신 모듈 | 전달 데이터 | 용도 |
| --- | --- | --- |
| 그래프 구축 (C-01~C-05) | 청크 목록 + 메타데이터 | 엔티티/관계 추출, 온톨로지 매핑 |
| 임베딩 엔진 | 청크 텍스트 + context_prefix | 벡터 임베딩 생성 및 인덱싱 |
| 검색 모듈 (R-01~R-04) | 메타데이터 | 필터 검색, 원본 추적 |
| 운영 모듈 (O-01) | 파싱/청킹 처리 토큰 수 | 토큰 비용 산출 |
| 운영 모듈 (O-03) | 파이프라인 처리 상태 | 진행률 모니터링, 실패 건 재시도 |

### 7-2 증분 업데이트 연동 (O-04)

- 문서 재업로드 시 변경된 청크만 식별하여 후속 모듈에 부분 업데이트 요청
- 변경 감지 기준: 청크 해시(hash) 비교

## 8. 수용 기준 (Acceptance Criteria)

### D-01 멀티 포맷 파싱

- [ ] PDF, DOCX, HWP, XLSX, CSV 5종 포맷의 샘플 문서(포맷별 5건)를 업로드하여 텍스트·표가 정상 추출됨을 확인
- [ ] 스캔 PDF(OCR 대상) 5건에 대해 한국어 텍스트 인식률 95% 이상 달성
- [ ] 비밀번호 보호 파일, 손상 파일 업로드 시 명확한 오류 메시지 반환
- [ ] 100MB 파일 업로드 및 파싱이 정상 완료됨

### D-02 의미론적 청킹

- [ ] 동일 주제의 문장이 서로 다른 청크로 분리되지 않음을 수동 검증 (샘플 5건)
- [ ] 청크 크기가 설정된 범위(300~800 토큰) 내에 90% 이상 포함
- [ ] 표가 행·열 구조를 유지한 채 독립 청크로 생성됨
- [ ] 상위 헤딩 정보가 context_prefix에 정확히 반영됨

### D-03 메타데이터 추출

- [ ] 파일 내장 메타데이터(작성자, 생성일, 제목)가 정상 추출됨 (샘플 10건)
- [ ] LLM 기반 문서 유형 자동 분류 정확도 85% 이상 (샘플 30건)
- [ ] 추출된 메타데이터로 필터 검색 시 정확한 결과 반환

| 리스크 | 영향 | 대응 방안 |

| HWP 파싱 라이브러리 성숙도 부족 | 표·글상자 등 복잡 구조 파싱 실패 가능 | 한컴 공식 API 또는 hwp5 오픈소스 라이브러리 사전 검증, 필요 시 HWP→PDF 변환 후 파싱 병행 |
| 스캔 품질 저하 | OCR 인식률 저하 | 이미지 전처리(디스큐, 노이즈 제거) 파이프라인 추가, 인식 신뢰도 임계값 이하 시 수동 검수 큐 전달 |
| 대용량 문서 처리 지연 | 사용자 대기 시간 증가 | 비동기 처리 + 진행률 표시, 완료 시 알림 발송 |
| 의미론적 청킹 품질 불안정 | 검색 성능 저하 | 청킹 결과에 대한 사용자 피드백 수집 및 파라미터 튜닝 루프 구축 |
| LLM 기반 메타데이터 추출 비용 | 대량 문서 처리 시 토큰 비용 증가 | 규칙 기반 추출 우선, LLM은 보완용으로 활용. 토큰 사용량 모니터링(O-01) |
